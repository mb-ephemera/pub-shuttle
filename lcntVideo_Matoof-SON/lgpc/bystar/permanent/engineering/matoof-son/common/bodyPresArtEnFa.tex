\pnote{

Greetings.

This is Mohsen Banan.

I am a software and telecom and internet engineer.

The title of this presentation is:
``soncli.mcs: An After-Market Open-Source Enabler''
``An Evolutionary Bridge From Nokia EdenNet SON to More Than SON''

Nokia EdenNet SON is a mature and widely deployed Self-Organizing Network platform.
It is a powerful system --- but it operates inside the Telecom Walled Garden,
and its capabilities are locked behind a single Web UI.

This presentation is about breaking that lock.

Think of SON Modules as CDs and the EdenNet SON Web UI as a CD-Player.
soncli.mcs is a second CD-Player --- one that you control, that runs on the command line,
and that you can build upon freely.

The rest of this presentation shows how soncli.mcs is implemented, what it enables,
and where it can take you.

}

\end{frame}

    \begin{exampleblock}{Act I: soncli.mcs as an After Market Story.}
    \end{exampleblock}

    \begin{columns}[T]
\column{0.33\textwidth}

\textbf{Vendor \& Product}
\begin{itemize}
  \item Nokia EdenNet SON
\end{itemize}

\column{0.33\textwidth}

\textbf{Customers \& Users}
\begin{itemize}
  \item T-Mobile, BT, Orange, Telstra, NTT DOCOMO, ...
\end{itemize}

\column{0.33\textwidth}

\textbf{Gap \& Our Offerings}
\begin{itemize}
  \item soncli.mcs -- A CLI Interface for Running SON Modules
\end{itemize}

\end{columns}

      \begin{block}{Act II: Absorb EdenNet SON Into the Much Richer BISOS-PyCS Framework}
        Providing: Web-UI for soncli.mcs, Observability, Parallel Execution
      \end{block}

Once Absorbed, the soncli.mcs derivative inherits:

\begin{itemize}
  \item Web UI for soncli.mcs for GUI specification of parameters and targets
  \item Observability of execution and results
  \item Nation Wide Parallel Execution of Modules
\end{itemize}


\pnote{

  This story has 4 Acts. Act I is about soncli.mcs being an aftermarket solution.
  Aftermarket dynamics involve: 1) a product, in this case EdenNet-SON 2) a vendor, Nokia,
  3) customers, T-Mobile, NTT-DOCOMO, and around 20 others. 4) a gap in the original
  offerings, in this case the ability to configure and run modules on command line.
  Filling this gap is what soncli.mcs is about. EdenNet-SON is a mature product (10+
  years), so why does this gap still exist. My conjectures are: first Nokia wrongly
  perceives that ability as detrimental to its interests; second, the telecom users of
  SON are not sufficiently aware that something like soncli.mcs can enable them to
  do a whole range of things on their own; third, the Telecom Walled Garden prevents others
  from providing solutions. In this case, by wrapping soncli.mcs into a larger
  Open Source context we are bypassing the Telecom Walled Garden. That larger Open Source
  is called BISOS-PyCS.

  The power of soncli.mcs comes from it being an enabler. You can build on it. In Act II,
  we add to soncli.mcs a general purpose Open Source Web UI called the PyCS-Player and add
  observability to it through Grafana and parallel executions to it through AirFlow.
  Now all that surrounds soncli.mcs is Open Source and can be expanded.

}

  Act III is about extending the limited abstractions of EdenNet SON's ``Modules'' and ``Targets'' beyond RAN and beyond Telecom.

\textbf{Nokia EdenNet SON Parts List}

\begin{itemize}
  \item \textbf{SON Python Modules}
    \begin{itemize}
      \item SON Python \textbf{Custom} Modules
      \item SON Python \textbf{Nokia} Modules
    \end{itemize}
  \item \textbf{SON Python API}
  \item \textbf{SON Python Software Development Kit (SDK)}\\
    {\color{darkred}  \textit{(The SDK is also an External Execution Facility --- EEF)}}
  \item SON Web User Interface
  \item SON Execution Engine
  \item SON OSS Interface Adapters
\end{itemize}

\pnote{
  Nokia EdenNet SON is composed of several distinct parts, and it helps to understand
  what each one does before we talk about soncli.mcs.

  At the core are SON Python Modules. These are pieces of Python code that implement
  specific network management tasks. Some modules are provided by Nokia; others are
  written by the operator's own team.

  Modules interact with the network through the SON Python API --- a published interface
  that abstracts the underlying network details.

  The SON Software Development Kit, or SDK, is the set of Python libraries that
  implement this API. The SDK communicates with the EdenNet platform over a remote
  interface. Importantly, the SDK is not just a development tool --- it is also an
  External Execution Facility. That distinction is key to understanding soncli.mcs.

  On top of all this sits the SON Web User Interface --- the single player that Nokia
  provides for running modules. Beneath it are the Execution Engine and the OSS
  Interface Adapters that connect to the radio network.

  soncli.mcs touches only three of these parts: the Modules, the API, and the SDK.
  It needs nothing else from Nokia.
}

\pnote{
  This figure shows the Nokia EdenNet SON ecosystem as a whole.

  On the left is the Nokia EdenNet SON Platform --- a closed, proprietary system.
  It includes the Web UI, the execution engine, the OSS adapters, and the platform
  libraries. Everything runs inside Nokia's controlled environment.

  On the right is what we call the SON External Execution Platform. This is where
  soncli.mcs lives. The only bridge between the two sides is the SDK Remote API ---
  the same interface that Nokia's own Web UI uses internally.

  soncli.mcs is an External Modules Player. It sits entirely outside the Nokia platform
  and communicates with it only through that published SDK interface.
  Nokia does not need to be involved, approve, or even be aware.

  This is the architectural foundation of the aftermarket story.
}

\begin{small}
\begin{columns}[T]
\column{0.5\textwidth}

\begin{center}
        \colorbox{blue!30}{\color{black} \textbf{Strength:}}
\end{center}

\begin{itemize}
  \item Choice of Python (12 years ago)
  \item Modules Architecture for RAN Systems Management
  \item Targets Architecture for Cells
  \item Python API for RAN and Telecom  Sys Mgmt Abstractions
\end{itemize}

\column{0.5\textwidth}

   \begin{center}
      \colorbox{blue!60}{\color{white} \textbf{Weaknesses:}}
    \end{center}

    \begin{itemize}
      \item Jailed in the Telecom Walled Garden
      \item Abstraction of Targets is Limited to Cells
      \item Limited in Scope to RAN and Telecom
      \item Limited to a a single Modules Player (Web-UI)
      \item No CLI for Modules Execution

    \end{itemize}

\end{columns}
\end{small}

\begin{exampleblock}{With soncli.mcs We Can Liberate Modules and Targets}
  \begin{itemize}
    \item Inside of the Telecom Walled Garden Open Up and Extend EdenNet SON
    \item Profit from the Nokia SON Experience of ``Modules and Target Oriented Open Fabric''
          (MaTOOF) for Systems Management Outside of the  Telecom Walled Garden
  \end{itemize}
\end{exampleblock}


\pnote{

  Let us be honest about both the strengths and the weaknesses of Nokia EdenNet SON.

  On the strength side, Nokia made some excellent architectural choices. Choosing
  Python over a decade ago was forward-looking. The Modules architecture for RAN
  systems management is clean and well-conceived. The Targets architecture for cells
  and the Python API for telecom abstractions are solid foundations.

  On the weakness side, the platform is jailed inside the Telecom Walled Garden.
  The abstraction of Targets is limited to cells. The scope is confined to RAN and
  Telecom. There is only one Modules Player --- the Nokia Web UI. And there is no
  CLI for module execution.

  soncli.mcs directly addresses these weaknesses. It provides the missing CLI.
  It opens the door to multiple players. And by bringing EdenNet SON into the
  BISOS-MaTOOF framework, it extends the concepts of Modules and Targets far
  beyond what Nokia originally envisioned.

  The strengths of the Nokia architecture are preserved. The weaknesses are overcome.

}

\end{frame}

\pnote{
  This figure shows the layered architecture of the PyCS framework.

  PyCS --- Python Command Services --- is the open source foundation on which
  soncli.mcs is built. It was designed specifically to support unified commands
  and services that can be executed both on the command line and through web interfaces.

  The framework is layered. At the bottom is BISOS --- the ByStar Internet Services
  Operating System --- a Debian-based substrate for building open source
  software-service continuums. PyCS sits on top of BISOS.

  On top of PyCS are domain-specific packages. For soncli.mcs, the relevant one is
  tocsModules --- Target Oriented Command Services Modules --- which provides the
  uploadable module and parallel execution capabilities.

  The key insight is that PyCS was designed with exactly the abstractions that
  Nokia EdenNet SON uses --- modules and targets. This is why absorbing soncli.mcs
  into PyCS requires so little code.

\pnote{
  Nokia EdenNet SON is composed of several distinct parts, and it helps to understand
  what each one does before we talk about soncli.mcs.

  At the core are SON Python Modules. These are pieces of Python code that implement
  specific network management tasks. Some modules are provided by Nokia; others are
  written by the operator's own team.

  Modules interact with the network through the SON Python API --- a published interface
  that abstracts the underlying network details.

  The SON Software Development Kit, or SDK, is the set of Python libraries that
  implement this API. The SDK communicates with the EdenNet platform over a remote
  interface. Importantly, the SDK is not just a development tool --- it is also an
  External Execution Facility. That distinction is key to understanding soncli.mcs.

  On top of all this sits the SON Web User Interface --- the single player that Nokia

\pnote{
  Nokia EdenNet SON is composed of several distinct parts, and it helps to understand
  what each one does before we talk about soncli.mcs.

  At the core are SON Python Modules. These are pieces of Python code that implement
  specific network management tasks. Some modules are provided by Nokia; others are
  written by the operator's own team.

  Modules interact with the network through the SON Python API --- a published interface
  that abstracts the underlying network details.

  The SON Software Development Kit, or SDK, is the set of Python libraries that
  implement this API. The SDK communicates with the EdenNet platform over a remote
  interface. Importantly, the SDK is not just a development tool --- it is also an
  External Execution Facility. That distinction is key to understanding soncli.mcs.

  On top of all this sits the SON Web User Interface --- the single player that Nokia
  provides for running modules. Beneath it are the Execution Engine and the OSS
  Interface Adapters that connect to the radio network.

  soncli.mcs touches only three of these parts: the Modules, the API, and the SDK.
  It needs nothing else from Nokia.
}

\pnote{
  Let us be concrete about the immediate and tangible benefits of soncli.mcs.

  For SON module developers, the most immediate benefit is the ability to load and
  run modules from the command line. This means you can subject your modules to
  regression testing and build CI/CD pipelines for continuous verification ---
  things that are essentially impossible with the Nokia Web UI alone.

  For AI-assisted development, soncli.mcs is a prerequisite. Agentic AI tools
  need a CLI interface to iteratively test and refine code. Without soncli.mcs,
  AI cannot close the loop on SON module development.

  For operations teams, soncli.mcs enables nationwide execution of any SON module
  across all servers simultaneously, with results collected centrally.

  And because soncli.mcs sits inside the BISOS-MaTOOF framework, the scope of
  what modules can target extends beyond cells and RAN --- into broader systems
  management.

  Finally, with soncli.mcs, the path toward multiple SON module players is open.
  The ecosystem becomes richer for everyone.

  The SDK already permits external execution --- soncli.mcs simply makes that
